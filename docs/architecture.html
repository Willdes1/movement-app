01 — Orientation

What the system is

Atlas Prime is a single Next.js application. The same codebase renders the user-facing pages and hosts the backend as serverless API routes — there is no separate backend server. Everything persistent lives in one Supabase project (a Postgres database with authentication, file storage, and realtime built in). The heavy thinking — writing training plans, coaching cues, and voices — is delegated to outside AI providers and paid per use.

The mental model: the browser holds a login token, the API routes are the gatekeepers, Postgres is the single source of truth, and the AI providers are contractors the routes hire job-by-job. The rest of this document follows the data through each of those.

Hard constraint

Vercel's Hobby plan kills any function at 60 seconds. Every bulk job (voicing, library fill, video curation) is therefore chunked into small parallel batches and looped from the client — never one long server call.

03 — The core pattern

How one request actually flows

This is the single most important pattern in the backend. Every coach and admin route follows it. There are two database connections, used for two different jobs:

Browser Holds the user's login token (JWT) anon client
→
API route Receives the token in the request header /api/…
→
① Verify caller anon client + getUser() proves who they are identity
→
② Service role Trusted key that bypasses row security server only
→
Postgres Reads / writes the actual data source of truth

Why two clients?

The anon client (lib/supabase.ts) runs in the browser and is bound by Row-Level Security (RLS) — it can only touch rows the logged-in user owns. The service-role client only ever exists inside an API route, after the caller's identity is proven; it holds a secret key that bypasses RLS so the route can do trusted work (read another user's program, write cost logs, run an admin job). The rule: verify with the anon key, then act with the service key. Mixing them up is the classic security mistake this pattern prevents.

Real example from this codebase

Cost logging silently failed for months because logTokens() wrote with the anon key to an RLS-locked table — every insert was rejected and swallowed. The fix was to use the service-role key. Same pattern, correctly applied.

04 — Who is who

Identity, roles & impersonation

Every account is a row in profiles. A single role column decides which surface a person sees. Sign-up sets it: the athlete path defaults to a normal user, the coach path (/coaches) stamps role = coach, and promo codes can grant any role.

freeAthlete app (default)signup default

The effectiveUserId rule

AuthContext exposes two identities: user (the real logged-in account) and effectiveUserId (whose data to read/write). They're normally the same. When an admin uses Zoom In to support a client, effectiveUserId swaps to that client so the admin sees exactly their experience — while user stays the admin, and every action is written to an audit log. All data reads use effectiveUserId, never user.id.

06 — Flagship flow

The AI Plan Engine (APIE)

When an athlete generates a plan, /api/generate-plan runs a multi-agent pipeline. It first retrieves relevant coaching knowledge from a vector search, then passes a draft plan through a chain of specialist Claude agents, each rewriting its own slice. Any agent that fails is skipped — the plan still returns.

RAG retrieval Embed the athlete's profile, search the 784-item knowledge store (pgvector) embeddings
→
S&C Agent Drafts the 7-day plan from retrieved knowledge Claude
→
PT / Rehab Safety pass — swaps contraindicated moves. Has veto Claude
→
Sports Specialist Rewrites warmups + coaching to the sport Claude
→
Mobility Owns morning / cooldown / evening blocks Claude
→
Mindset Adds the warrior-philosophy layer Claude
→
Recovery Rest-day programming Claude
→
7-day plan Saved to training_programs Postgres

The knowledge store is the differentiator: real coaching principles, sport protocols, and rehab guidance embedded as vectors, retrieved by similarity so the agents write from grounded material rather than guesswork. Every agent call is metered (see Cost Tracking).

07 — Coach ↔ athlete

Coached Mode

When a coach takes on an athlete, the coach's program takes over the athlete's app — the AI plan pauses and the coach's calendar becomes the main experience. The handoff is explicit and reversible.

Layered on top: real-time messaging (Supabase Realtime), per-client notes, compliance analytics, in-person session logging by the coach, and optional voice cloning — the coach records a sample, ElevenLabs mints a voice, and the athlete's read-aloud button speaks cues in the coach's own voice (cached per exercise). Pausing coaching resurfaces the athlete's AI plan; nothing is lost.

08 — Filling the library

The content engine

The exercise library is preloaded so any athlete worldwide gets a zero-lag plan. One admin tool feeds two downstream pipelines — text is written by Claude, then voiced by OpenAI and matched to video.

Built-in guards keep it cheap: the builder remembers saturated categories and skips them with zero token cost; voicing runs a hands-free loop (batches of 20, 8-wide) until the whole backlog is done; the upgrade pass finds stragglers by staleness so it never re-charges for finished rows.

09 — Following the money

Cost tracking

Every route that spends money on an AI provider logs the exact dollar cost. Because providers bill differently — Claude per token, OpenAI per character (TTS), per token (embeddings) or per minute (Whisper) — each route computes its own cost and stores the dollar figure. The Spend Tracker just sums the stored amounts, so mixed providers roll up accurately.

12 — Guardrails

Security model

GuardrailWhat it does
Row-Level SecurityPostgres enforces per-row ownership on every table — the browser's anon key can't read another user's data.
Service-role splitTrusted writes happen only inside API routes, after the caller's JWT is verified. The secret key never reaches the browser.
protect_admin_roleA database trigger blocks any update that would change an admin's role away from admin — even the coach-signup flow can't demote an admin.
SECURITY DEFINER fnsCross-role reads go through controlled functions instead of recursive RLS policies (which once caused an admin lockout).
admin_permissionsPartner admins get scoped, per-section grants — they are not is_admin and can't see owner-only data.
Impersonation auditEvery admin Zoom-In action is written to an audit trail; the real actor is always recorded.
Verified writesMoney/state writes read the row back and assert it persisted; a failed write logs a loud [VERIFY_FAIL] + a harness_events row and throws — never silent (lib/verified-write.ts).
TelemetryThe app's own safety checks (verify / route / migration / smoke) are recorded to harness_events and surfaced in the Telemetry tab; route errors forward to Sentry when a DSN is configured.

master is production — no staging yet (documented in docs/harness-staging.md, deferred until post-launch). Database changes are the one manual step (SQL run by hand in Supabase) — but now tracked: every migration self-registers in the applied_migrations ledger, and the Telemetry Migrations panel flags any file you forgot to run. After a deploy, the smoke suite (Telemetry ▶ Run, or npm run smoke-test) exercises the critical paths and asserts real DB side effects. Secrets live only in Vercel env vars, never in the repo. Health, spend & telemetry are watched from the Admin Portal.
